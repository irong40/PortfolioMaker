## What Was Built
Three features added in one session: (1) Panorama detection and auto-stitching using OpenCV for DJI drone panorama photo sets, (2) Rename from "Portfolio Maker" to "Sortie" across all source files, (3) Critical fix to MipMap Gaussian Splat integration — task JSON was sending empty camera/image metadata causing exit code 9015. Also added settings persistence (save/restore UI state on close/open) and a Reset button.

## Key Decisions

### Panorama Detection
- Detection is folder-structure-based (PANORAMA/ subdir with PANO_*.JPG files), not EXIF-based — DJI M4E has no panorama mode flag in XMP metadata
- `scan_panoramas()` checks both current dir and parent dir for PANORAMA/ folder to handle SD card layout where PANORAMA/ is sibling to DJI_xxx/ photo folders
- Stitching uses `cv2.Stitcher_PANORAMA` with OpenCL disabled (caused GPU memory crashes)
- Images downscaled to 2000px width before stitching to avoid OOM on 25-35 photo sphere panos

### MipMap Task JSON Fix
- Root cause: `camera_meta_data` and `image_meta_data` were empty arrays — MipMap needs full camera calibration and per-image GPS/orientation
- Camera intrinsics extracted from DJI's `DewarpData` XMP field, reordered from DJI format (k1,k2,p1,p2,k3) to MipMap SDK format (k1,k2,k3,p1,p2)
- RTK GPS auto-detected from XMP `GpsStatus` field — sets `pos_sigma` to [0.03, 0.03, 0.06] instead of [2, 2, 5]
- Gimbal pitch/roll/yaw converted to 3x3 rotation matrix via ZYX Euler convention
- Validated output matches known-good task JSON from MipMap Desktop GUI (D:\ProjectErrors\TaskLog\at_task.json)

### Settings Persistence
- Simple JSON file (`sortie_settings.json`) in app directory
- Saves on window close (WM_DELETE_WINDOW protocol + Quit button)
- Loads on startup with safe defaults for missing keys
- Reset button in Advanced section clears file and resets all UI state

### Rename
- "Sortie" chosen for triple meaning: military sortie (drone ops), sort (what app does), sortie out (bring back deliverables)
- Old files kept (portfolio_maker.py, .pyw, .ico) for backwards compatibility — should be cleaned up
- sortie.py is a copy of portfolio_maker.py (not a symlink) — tech debt

## Areas of Concern
1. **No tests for any new code** — panorama detection, stitching, MipMap metadata extraction, settings persistence all lack unit tests. This is the biggest gap.
2. **sortie.py is a duplicate of portfolio_maker.py** — changes must be synced manually. Should delete one and have the other import.
3. **Gimbal-to-rotation-matrix math** — the ZYX convention was implemented from first principles, not validated against a reference implementation. The output looks reasonable but edge cases (gimbal lock, extreme roll) are untested.
4. **UTM zone hardcoded to 32618** (zone 18N) in mipmap_service.py — only correct for US East Coast. Will fail for users in other regions.
5. **Panorama stitching runs synchronously in sort_photos()** — on large datasets with many pano sets, this could block for minutes with no progress feedback to the GUI.
6. **NodeODM integration uses NodeODM direct API** (`/task/new`), not WebODM API (`/api/projects/{id}/tasks/`). Works locally but won't work for WebODM setups with authentication.

## Known Limitations
1. Panorama stitching fails on some photo sets (low texture, repetitive patterns) — OpenCV limitation, not a bug
2. MipMap integration not yet tested end-to-end with fixed task JSON (exit 9015 was the blocker, now fixed but not re-run)
3. No WebODM API authentication support — NodeODM direct only
4. generate_icon.py:18 still has one stale "Portfolio Maker" reference in docstring
5. Camera parameter validation: if DewarpData is malformed or missing, fallback uses estimated focal length which may produce poor AT results
